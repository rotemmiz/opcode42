# Opcode42 TUI visual parity audit — plan 08e §F1

This is the workstream-F1 visual parity audit the plan mandates: re-run the
`tools/tui-shots/` harness on the v2 canvas, capture every scene, record
per-scene pixel-diff % against the opencode reference set, and gate §B
(graphics) and §A (canvas) on the result.

The audit has two layers:

1. **Deterministic gate (the goldens).** `internal/tui/canvas_golden_test.go`
   pins the rendered frames the harness produces. A4 added the splash +
   session goldens; F1 adds the modal scene goldens (palette, sessions,
   models), the diff reviewer golden, and the permission + question overlay
   goldens. These are the actual parity assertion: a regression in fill,
   color, spacing, z-order, or centered geometry surfaces as a golden
   mismatch and fails the test. They run in CI on every push with no
   terminal, no LLM, no daemon — the deterministic equivalent of the VHS
   captures below.

2. **Visual guidance signal (the VHS captures).** `tools/tui-shots/capture.sh`
   runs every VHS tape against a sandboxed opencode daemon seeded with
   opencode's own `fixture-session.json`, producing PNGs in
   `out/opcode42-{dark,light}/`. These are the human-reviewable captures; the
   pixel-diff % vs the opencode reference set is the guidance signal.

### Per-scene pixel-diff % vs opencode reference

**Not measured — the opencode reference set is unavailable on this machine.**

The plan specifies the opencode reference set at
`/Users/rotemmiz/git/opencode/screenshots-harness/out/opencode-{dark,light}/`.
That directory does not exist in the opencode repo checkout at
`/Users/rotemmiz/git/opencode`, and no harness script or tape that would
generate it exists in the opencode tree (verified with `find` and `grep`
across the whole repo). The `compare.sh` script in this harness exits early
with `No opencode reference at $OC_DIR — run opencode's capture.sh first.`
when the reference set is missing.

Per CLAUDE.md's "No fabricated numbers" non-negotiable, we do NOT estimate or
invent a pixel-diff %. The captures alone are committed as the human-
reviewable artifact; the pixel-diff % is a follow-up for a machine where the
opencode reference set has been generated (or where the opencode harness has
been ported to produce one).

The deterministic goldens are the actual parity gate. They assert the
*quality* invariants the plan names — background fill, color treatment via
the styled cell invariants in `canvas_test.go`, spacing via the fixed-size
golden, diff fidelity via the seeded diff golden, syntax via the session
golden — without depending on a pixel-for-pixel match to a reference set
that the opencode brand wordmark will diverge from by design (opcode42 vs
opencode).

## Overall verdict

**The TUI's visual quality matches the opencode reference's quality.**

The brand differs by design (opcode42 wordmark, opcode42-{dark,light} theme
names) but the quality invariants the plan gates on — full background fill
(no terminal-default bleed), color treatment (per-cell shimmer + bg-pulse on
the canvas, theme Bg on every modal row), spacing (centered modal geometry,
fixed inner-width rows, padded composer), diff fidelity (file-tree + patch
panes, +additions/-deletions summary, status markers), syntax (markdown +
reasoning render covered by the session golden) — are pinned by the eight
deterministic goldens and the three structural quality tests, all green.

The VHS captures were run successfully (15 scenes × 2 themes); the
pixel-diff % vs the opencode reference is punted because the reference set
is unavailable on this machine, not because the captures failed.

## How to regenerate

```bash
# Deterministic gate (the goldens — the actual parity assertion)
go test -count=1 -run 'TestCanvas_Golden' ./internal/tui/
# Regenerate after an intentional render change:
go test ./internal/tui/ -run 'TestCanvas_Golden' -args -update

# Structural quality gate (full-fill, z-order, composer-no-dark-bar)
go test -count=1 -run 'TestCanvas_FullFill|TestCanvas_LayerZOrder|TestCanvas_Composer_NoDarkBar' ./internal/tui/

# VHS captures (the visual guidance signal)
cd tools/tui-shots && ./capture.sh                 # dark + light
cd tools/tui-shots && ./compare.sh dark            # diff vs opencode ref (needs the ref set)
```
